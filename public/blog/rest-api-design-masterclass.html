Part I of IV

Foundations & Resource Design

REST philosophy, HTTP internals, and the art of designing clean resource hierarchies. The fundamentals that every decision flows from.

The difference between a junior dev who builds APIs and a senior engineer who designs them is this: the senior understands that every URL, every status code, every field name is a contract — and contracts are hard to break.

What is REST? The Philosophy

REST (Representational State Transfer) was defined by Roy Fielding in his 2000 PhD dissertation. It's not a protocol, not a standard — it's an architectural style. Understanding this distinction is what separates junior devs from engineers.

The 6 REST Constraints

Every constraint exists for a reason. Internalize these — they drive every design decision you'll ever make.

🖥️

1. Client-Server Separation

Client handles UI. Server handles data. They evolve independently. Your mobile app, web app, and third-party integrations can all use the same API.

📦

2. Statelessness

Every request contains ALL information needed to process it. The server stores ZERO client session state. Auth token, pagination offset, user ID — all in the request.

💾

3. Cacheability

Responses must declare whether they're cacheable. Cache reduces server load, cuts latency, and improves perceived performance. Cache-Control and ETag implement this.

🏛️

4. Uniform Interface

Resources identified by URIs. Manipulation through representations. Self-descriptive messages. HATEOAS. This is what makes REST RESTful.

🧅

5. Layered System

Client can't tell if it's connected directly to the server or through a load balancer, CDN, or proxy. Enables infrastructure changes without breaking clients.

📄

6. Code on Demand (Optional)

Server can extend client functionality by sending executable code. Optional constraint. Used rarely in practice but enables rich client experiences.

REST vs Other Paradigms

ParadigmBest ForWeakness
RESTCRUD APIs, public APIs, microservicesOver/under-fetching, multiple roundtrips
GraphQLComplex data graphs, client-driven queriesCaching complexity, N+1 problems
gRPCInternal microservices, low-latency streamingBrowser support limited, binary format
WebSocketReal-time bidirectional communicationStateful, scaling is hard
SOAPEnterprise, banking, legacy systemsVerbose XML, complex WSDL
💡 Senior Engineer Insight

REST is not "always the right answer." A senior engineer chooses the right tool. Real-time features? WebSocket. Internal service mesh? gRPC. Complex product queries? GraphQL. Know REST deeply, know when NOT to use it.

Resource Design — The Core Skill

Resource design is THE skill that defines API quality. A well-designed resource model makes an API intuitive. A bad one creates technical debt that haunts you for years.

Think in Nouns, Not Verbs

❌ BAD — RPC-style

POST /createUser
POST /deleteUser
POST /getUserById
POST /updateUserEmail
POST /activateUserAccount
  
✅ GOOD — Resource-oriented

POST   /users
DELETE /users/{id}
GET    /users/{id}
PATCH  /users/{id}
PATCH  /users/{id}/status

Resource Hierarchy

Users     └─ /users/{userId}/addresses    # owned by user
          └─ /users/{userId}/orders

Products  └─ /products/{id}/reviews
          └─ /products/{id}/images

Orders    └─ /orders/{id}/items
          └─ /orders/{id}/payments
          └─ /orders/{id}/shipments
⚠️ Nesting Rule of Thumb

Never nest more than 2 levels deep. /a/{id}/b/{id} is fine. /a/{id}/b/{id}/c/{id}/d is a red flag. Deep nesting creates tight coupling and fragile URLs. If you need deeper, create a top-level resource.

The Action Pattern

Some operations are inherently procedural. Force-fitting them into CRUD creates confusion. Use sub-resources as actions:

POST /orders/{id}/cancel          # Cancel order (state machine)
POST /orders/{id}/refund           # Issue refund
POST /users/{id}/verify-email      # Trigger verification flow
POST /payments/{id}/capture        # Capture pre-auth payment

URL Architecture — Naming Rules

❌ BAD patterns

 /getUsers
 /Users
 /user_list
 /GetAllUsers
 /Users/{UserId}/Orders
 /USER-PROFILE
  
✅ GOOD patterns

 /users
 /users
 /users
 /users
 /users/{userId}/orders
 /users/{userId}/profile
  • Use lowercase always. Never camelCase or PascalCase in URLs.
  • Use hyphens for multi-word paths: /blog-posts not /blog_posts
  • Use plural nouns for collections: /users not /user
  • Use camelCase for query params: ?sortBy=createdAt&pageSize=20
  • Prefer UUIDs over sequential IDs — sequential IDs reveal scale and enable enumeration attacks

Query Parameter Taxonomy

# FILTERING
GET /products?category=electronics&inStock=true&minPrice=100

# SORTING
GET /users?sortBy=createdAt&order=desc

# PAGINATION (cursor-based preferred)
GET /users?cursor=eyJpZCI6MjB9&limit=20

# FIELD SELECTION — reduce payload
GET /users?fields=id,email,name
GET /orders?include=items,customer    # expand related

Request & Response Design

The Envelope Pattern

Always wrap responses. Never return a naked array. The envelope allows you to add metadata without breaking clients.

# GET /users — collection response
{
  "success": true,
  "data": [{
    "id":        "usr_01HX7K...",
    "email":     "alice@example.com",
    "firstName": "Alice",
    "createdAt": "2024-01-15T10:30:00Z"   // ISO 8601, always UTC
  }],
  "pagination": {
    "total": 1847, "page": 1, "pageSize": 20,
    "hasNext": true, "nextCursor": "eyJpZCI6IjIwIn0="
  },
  "meta": { "requestId": "req_abc123", "version": "v1" }
}

Data Type Rules

  • IDs: Strings, not integers. Use UUID v4 or prefixed IDs like usr_01HX7K. Never expose sequential DB IDs.
  • Timestamps: ISO 8601 UTC with Z suffix: "2024-01-15T10:30:00Z". Never Unix timestamps in JSON bodies.
  • Money: Integers (cents). "amount": 4999 = $49.99. Never floats — float math breaks money.
  • Enums: SCREAMING_SNAKE_CASE: "status": "PENDING_PAYMENT". Document all valid values.

Core Design Decisions

Status codes, authentication, versioning, pagination, and error handling — the decisions that define the quality of your API contract.

A junior dev returns 200 for everything and parses the body to find errors. A senior engineer uses HTTP exactly as it was designed: status codes are the contract, response bodies are the detail.

HTTP Status Codes — The Complete Guide

2xx — Success

200
OK
GET/PATCH success. Resource returned in body.
201
Created
POST success. Include Location header + created resource.
202
Accepted
Async processing started. Return job/status URL.
204
No Content
DELETE success. Empty body — nothing to return.

4xx — Client Errors

400
Bad Request
Invalid input, malformed JSON, validation failed.
401
Unauthorized
Not authenticated. Include WWW-Authenticate header.
403
Forbidden
Authenticated but not authorized for this resource.
404
Not Found
Resource doesn't exist. Also use to hide 403s.
409
Conflict
Duplicate email, version conflict, business rule violation.
422
Unprocessable
Syntactically valid but semantically wrong. Use for validation errors.
429
Too Many
Rate limited. Always include Retry-After header.

5xx — Server Errors

500
Server Error
Unhandled exception. Log it. Never expose stack traces.
502
Bad Gateway
Upstream service returned invalid response.
503
Unavailable
Maintenance/overload. Include Retry-After header.
504
Timeout
Upstream timed out. Check your service dependencies.
🚫 The Anti-Pattern That Kills APIs

Never return 200 OK with {"success": false, "error": "not found"}. This forces clients to parse every response body to know if the request succeeded. HTTP status codes exist exactly to avoid this. Use them.

Authentication & Authorization

🔑

API Keys

Server-to-server. Prefix: sk_live_.... Hash and store — never plain text. Support key rotation. Rate limit per key.

🎫

JWT (Access + Refresh)

Stateless user sessions. Access token: 15min. Refresh: 30d, stored server-side for revocation. Never put sensitive data in payload — it's base64, not encrypted.

🔐

OAuth 2.0

Third-party access delegation. Four flows: Authorization Code (web), PKCE (mobile/SPA), Client Credentials (server-to-server), Device Flow.

JWT Flow — Under the Hood

# 1. LOGIN
POST /auth/login  →  {"accessToken": "eyJ...", "refreshToken": "dGhp...", "expiresIn": 900}

# 2. USE on every request
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...

# 3. REFRESH before expiry
POST /auth/refresh  {"refreshToken": "dGhp..."}  →  {"accessToken": "eyJ_new..."}

# 4. LOGOUT — invalidate refresh token server-side
POST /auth/logout  →  204 No Content

# JWT payload — include what you need for auth decisions
{
  "sub":         "usr_01HX7K",
  "role":        "admin",
  "permissions": ["users:read", "users:write"],
  "orgId":       "org_company123",   // tenant ID for multi-tenant
  "exp":         1703002500             // 15 min from now
}

API Versioning Strategies

URL Path ✅ Most Common

/api/v1/users → /api/v2/users. Highly visible. Easy to test in browser. Cache-friendly. The industry default.

Custom Header

API-Version: 2024-01-01. Date-based (Stripe/Azure style). Clean URLs. Invisible in browser. Good for internal APIs.

Query Parameter

/users?version=2. Easy to implement but awkward for clients. Generally avoid unless you have legacy constraints.

Content Negotiation

Accept: application/vnd.myapi.v2+json. Most RESTfully correct. Extremely complex. Almost never used in practice.

❌ BREAKING — needs new version

- Removing an endpoint
- Removing a field from response
- Changing field type (string → int)
- Changing field name
- Adding required request field
- Changing status code behavior
  
✅ NON-BREAKING — safe to ship

- Adding new endpoints
- Adding optional response fields
- Adding optional request fields
- Adding new enum values (with care)
- Bug fixes, performance improvements
- Adding new query parameters

Pagination, Filtering & Sorting

Offset Pagination

?page=3&pageSize=20. Jump to any page. Easy to implement. Items shift if data added. Expensive for large offsets (LIMIT 20 OFFSET 10000 is slow).

Cursor Pagination ✅

?cursor=eyJpZCI6MjB9&limit=20. Consistent even with inserts/deletes. O(1) DB query. Works at infinite scale. Can't jump to page 7.

Keyset Pagination

?afterId=usr_123&limit=20. Uses resource's natural ID. Simple. Good for append-only feeds sorted by creation time.

# Cursor pagination response
{
  "data": [...],
  "pagination": {
    "limit":      20,
    "hasMore":    true,
    "nextCursor": "eyJpZCI6IjQwIn0="  // base64(JSON) of last item's sort fields
  }
}

Error Handling — The Complete System

Errors are part of your API's public interface. A well-designed error response tells the client exactly what went wrong and, when possible, how to fix it.

# 422 Unprocessable Entity — Validation error
{
  "success": false,
  "error": {
    "code":      "VALIDATION_ERROR",
    "message":   "Request validation failed",
    "details":   [
      { "field": "email",    "code": "INVALID_FORMAT" },
      { "field": "password", "code": "TOO_SHORT", "meta": {"minLength": 8} }
    ],
    "requestId": "req_abc123",
    "docsUrl":   "https://docs.api.com/errors/VALIDATION_ERROR"
  }
}
AUTH:        UNAUTHORIZED → 401  |  TOKEN_EXPIRED → 401  |  FORBIDDEN → 403
RESOURCE:    NOT_FOUND → 404     |  ALREADY_EXISTS → 409  |  VERSION_CONFLICT → 409
VALIDATION:  VALIDATION_ERROR → 422  |  INVALID_FORMAT → 422
BUSINESS:    INSUFFICIENT_FUNDS → 422  |  ORDER_NOT_CANCELLABLE → 422
RATE:        QUOTA_EXCEEDED → 429
SERVER:      INTERNAL_ERROR → 500  |  SERVICE_UNAVAILABLE → 503

Advanced Topics

Security, rate limiting, caching, documentation, and the design principles that separate good APIs from great ones.

The difference between a good API and a production-grade API isn't the happy path — it's security, observability, graceful degradation, and documentation that makes developers love you.

Security Patterns — OWASP API Top 10

  • BOLA (Broken Object Level Auth) — Always verify the authenticated user owns the resource. GET /orders/456 must check order 456 belongs to this user.
  • Broken Auth — Short-lived JWTs, rotate secrets, store refresh tokens hashed, implement token revocation.
  • Excessive Data Exposure — Never return a DB row directly. Serialize only needed fields. That password_hash must never appear in any response.
  • Lack of Resource Limiting — Rate limit by IP AND by user. Limit response size. Limit query complexity in filters.
  • Mass Assignment — Never bind request body directly to DB models. Explicitly whitelist allowed fields. Never let users set role: "admin" on themselves.
  • SQL/NoSQL Injection — Always use parameterized queries. Never interpolate request values into queries.
  • SSRF — If your API fetches URLs from user input, validate and whitelist allowed domains.

Essential Security Response Headers

Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Content-Security-Policy: default-src 'none'

# CORS — tightly configure, never use *
Access-Control-Allow-Origin: https://yourfrontend.com
Access-Control-Allow-Methods: GET, POST, PATCH, DELETE, OPTIONS
Access-Control-Max-Age: 86400

Rate Limiting & Throttling

Token Bucket

Bucket refills at constant rate. Allows bursts up to capacity. Best for user-facing APIs — natural, human-friendly behavior.

Sliding Window

Counts in rolling time window. Prevents boundary attacks. Best accuracy. What most production APIs use.

Fixed Window

Simple counter per time window. Easy to implement. Vulnerable to 2× burst at window boundary. Good for internal limits only.

Leaky Bucket

Processes at constant rate. Smoothest traffic. Good for protecting downstream services that can't handle bursts.

# Rate limit headers on EVERY response (not just 429s)
X-RateLimit-Limit:     1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset:     1703005200  # Unix timestamp

# 429 Too Many Requests response body
{
  "error": {
    "code":       "RATE_LIMIT_EXCEEDED",
    "retryAfter": 3600,
    "upgradeUrl": "https://yourapi.com/pricing"
  }
}
✅ DX Best Practice

Include rate limit headers on every response, not just on 429s. Good API clients use these to proactively back off before hitting limits — dramatically reducing wasted requests.

Caching Strategies

# Public, cacheable for 1 hour (CDN + browser)
Cache-Control: public, max-age=3600

# Private (user-specific), browser only
Cache-Control: private, max-age=300

# Never cache (auth endpoints, mutations)
Cache-Control: no-store

# Stale-while-revalidate — best UX pattern
Cache-Control: public, max-age=60, stale-while-revalidate=300
# Serve cached for 60s. 60-300s: serve stale but refresh in background.

# ETags — conditional requests (save bandwidth)
Server sends:  ETag: "v1-hash-abc123"
Client sends:  If-None-Match: "v1-hash-abc123"
Unchanged →    304 Not Modified  (empty body)
Changed →      200 OK            (new content + new ETag)
Endpoint TypeCache StrategyTTL
GET /products (catalog)public + CDN5–60 min
GET /users/me (profile)private, browser only5 min
GET /config (settings)public + CDN1–24 hours
POST /auth/loginno-storenever
GET /search?q=...public, shared cache5–15 min

Senior Engineer Design Principles

Step 01
Domain Modeling
Start with nouns. Map your domain entities and their relationships before thinking about endpoints. Users, Products, Orders, Payments — define ownership and hierarchy first.
Step 02
User Flows
What does a client developer need to accomplish? Map the sequence of API calls for each user scenario. Identify where your design creates unnecessary friction.
Step 03
Resource Tree
Translate your domain model into resource hierarchies. Apply the nesting rules. Identify which resources need top-level endpoints vs nested access.
Step 04
Endpoint List
Write out every endpoint: method + path + description. For each, decide: auth required? Rate limited? Cacheable? Async or sync?
Step 05
Schemas
Write OpenAPI spec before writing code. Define every request/response shape. Have a frontend developer use your mock server before you build the backend.
Step 06
Review & Iterate
Walk through every decision: Is naming consistent? Are error codes helpful? Can this change without breaking clients in 2 years? Design is never finished on the first pass.
Senior engineers spend 10x more time designing than coding. An hour of good design prevents a week of refactoring.

Full Stack Data Flow — Kanban Board

USER OPENS BOARD

1. React makes parallel requests:
   GET /projects/{id}                                   ← project meta
   GET /projects/{id}/sprints?status=ACTIVE             ← active sprint
   GET /projects/{id}/tasks?sprint={sId}&fields=id,title,status,assignees

2. Server processes GET /tasks:
   a. Validate JWT → extract orgId, userId
   b. Check user has project access (Redis cache)
   c. Query PostgreSQL with indexes on (project_id, sprint_id, status)
   d. Serialize: map DB rows → API schema (exclude internal fields)
   e. Return Cache-Control: private, max-age=30

3. React renders Kanban board, groups tasks by status column

USER DRAGS TASK TO DIFFERENT COLUMN

1. Optimistic UI update (move card instantly in React state)
2. PATCH /tasks/{taskId}  {"status": "IN_REVIEW"}

3. Server processes:
   a. Validate JWT + task ownership/permission
   b. Validate state transition (TODO → IN_PROGRESS → IN_REVIEW → DONE)
   c. UPDATE tasks SET status='IN_REVIEW', updated_at=NOW()
   d. INSERT INTO task_activity (change log)
   e. Publish to Redis: {event: "task.status.changed", taskId}
   f. Return 200 with updated task

4. WebSocket server picks up Redis event →
   Broadcasts to all connected users in this project
5. Other team members' boards update in real-time
6. If PATCH fails → React rolls back optimistic update, shows toast error

Senior Engineer Self-Assessment

Can you check every one of these? That's your graduation benchmark.

01
Design first, code second. Write your OpenAPI spec before a single line of implementation.
02
Status codes are the contract. Never 200 with error in body. HTTP codes exist for a reason.
03
Idempotency matters. Know which operations are safe, which are idempotent. Design retries accordingly.
04
Version proactively. Decide versioning strategy before shipping v1. Deprecation windows must be communicated months in advance.
05
Security is not optional. BOLA checks, input validation, no excessive data exposure. Design with a threat model in mind.
06
Errors are a product feature. Structured error codes, field-level validation details, and docs URLs make developers love you.
07
Pagination from day one. No collection endpoint should ever return unbounded results. Cursor-based for anything at scale.
08
Rate limit headers everywhere. Not just on 429s — proactive clients use these to self-throttle before hitting limits.
🎯 The Secret of Senior Engineers

An hour of good API design prevents a week of refactoring. Every decision you make in your API design is a contract with your users. Write that contract carefully — because once published, you have to live with it.
